MDX is most valuable when it solves a real writing need, not when it turns every article into a miniature component demo.

## When to use Markdown and when to use MDX

A practical split looks like this:

- standard prose, lists, tables, and code blocks: use Markdown
- imported Astro components, local assets, or richer inline composition: use MDX

This theme supports both `.md` and `.mdx`, so the author can choose the lighter format by default and only opt into MDX when needed.

## Code blocks, highlighting, and copy buttons

Technical content needs more than plain paragraphs, so this article includes several common block types.

### Astro example

```astro
---
import MainLayout from "../layouts/MainLayout.astro";
import { getDictionary } from "../utils/i18n";

const locale = "en" as const;
const dictionary = getDictionary(locale);
---

<MainLayout title={dictionary.search.title}>
  <h1>{dictionary.search.title}</h1>
</MainLayout>
```

### TypeScript example

```ts
export function getTotalPages(totalItems: number, pageSize: number) {
  return Math.max(1, Math.ceil(totalItems / pageSize));
}
```

### Shell example

```bash
npm run check

```

The important part is not only that the code is highlighted, but that the demo proves the copy button remains practical on both desktop and touch devices.

## Why this article matters to the demo

If a theme claims to support Markdown and MDX but never shows imported components, local assets, or richer document structure, users still do not know whether it can carry real-world content.

This article exists to make those capabilities concrete, not theoretical.
