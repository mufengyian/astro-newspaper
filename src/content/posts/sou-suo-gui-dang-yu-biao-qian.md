当文章数量还很少时，首页几乎就是全部内容。但一旦内容开始积累，真正决定体验上限的，往往不是单篇文章模板，而是“找回内容”的能力。

## 搜索页负责解决“我知道自己想找什么”

这个主题的搜索不是全文数据库，而是构建期生成的静态索引。它默认把下面这些字段放进搜索数据里：

- 标题
- 摘要
- 分类
- 标签
- 正文纯文本
- 发布时间文案
- 阅读时长文案

对应的数据结构大致像这样：

```json
{
  "title": "搜索、归档与标签",
  "excerpt": "演示搜索索引与内容发现路径。",
  "category": "内容发现",
  "tags": ["Search", "Archive", "Tags"],
  "url": "/posts/sou-suo-gui-dang-yu-biao-qian/",
  "publishedLabel": "2026年3月22日",
  "readingMinutesLabel": "4 分钟阅读",
  "body": "当文章数量还很少时，首页几乎就是全部内容。"
}
```

这意味着你既可以搜标题，也可以搜正文里的概念词，还可以直接输入标签名。

## 归档页负责解决“我记得大概是那一年写的”

归档页按年份聚合文章，这在下列场景尤其有用：

1. 你知道内容不是最近写的，但不记得标题。
2. 你想回看某一阶段的项目记录。
3. 你想确认站点是不是形成了稳定输出节奏。

为了让这个能力不是空壳，这次 demo 内容刻意把文章发布时间分布在 2026、2025 和 2024 三个年份上。

## 标签页负责解决“我想沿着一个主题继续读”

标签与搜索不是替代关系，而是两种不同的阅读入口：

- 搜索更像直接查询
- 标签更像沿主题漫游

例如你读完这篇文章，如果对内容组织感兴趣，可以继续点这些主题：

- `Search`
- `Archive`
- `Tags`
- `SEO`

标签页还有一个额外价值：它会帮助相关文章模块更自然地工作，因为相关文章正是通过共享标签与分类进行打分的。

## RSS 解决的是“我不想每次都手动回来找”

一个结构完整的写作主题，不应该只考虑站内浏览，也要考虑站外订阅。RSS 在这里承担的是“把更新主动送出去”的角色。

如果你在 demo 里看到：

- `/rss/`
- `/rss.xml`
- `/en/rss/`
- `/en/rss.xml`

说明双语 feed 链路已经被完整跑通。

## 搜索、归档、标签和 SEO 其实是一条链

从用户视角看，这些能力像是分开的页面；但从内容系统视角看，它们共享的是同一份结构化内容数据。

| 能力 | 依赖字段 | 主要收益 |
| --- | --- | --- |
| 搜索 | 标题、摘要、正文、标签、分类 | 快速定位 |
| 归档 | 发布时间 | 时间维度浏览 |
| 标签 | 标签数组 | 主题维度浏览 |
| 相关文章 | 标签 + 分类 | 延长阅读路径 |
| RSS | 标题、摘要、URL、发布时间 | 站外订阅 |

## demo 为什么一定要把这些入口都做实

如果演示站只有几篇互不关联的短文，那么：

- 搜索几乎搜不到东西
- 标签页只有一两个标签
- 相关文章像随机推荐
- 归档没有年份层次

这也是为什么这次我会把 demo 文章做成一个彼此关联的内容组，而不是单独几篇“会显示就行”的孤岛文章。
