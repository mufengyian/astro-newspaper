技术主题最容易走向两个极端：

- 完全没有交互反馈，页面像打印稿
- 交互过多，正文反而成了背景板

这个主题的做法更偏中间：只保留那些对阅读、导航和轻量互动真正有帮助的部分。

## 预取与视图过渡

这个仓库已经启用了 Astro 的预取与视图过渡能力，所以你在站内点击文章、列表和导航时，会感觉页面切换更顺，但又不会像一个重 JavaScript 应用那样复杂。

这类能力最适合放在：

- 文章列表到文章页
- 归档页到详情页
- 站内导航之间

它们的价值在于“降低切换阻力”，而不是制造存在感。

## 代码复制

对技术写作来说，代码块复制按钮是很典型的“小功能、大收益”。如果每次复制命令都要手动选区，长篇教程的使用成本会立刻升高。

```bash
cp .env.example .env
npm install
npm run dev
```

## 回到顶部与阅读进度

这两项能力刚好分别服务于长文的两个阶段：

- 阅读进度：你正在读的时候，知道自己读到哪里
- 回到顶部：读完或跳读之后，能快速回到导航起点

它们都不需要抢眼，但必须稳定。

## 主题切换为什么不只是“配色偏好”

对于有大量代码块、表格和长文段落的页面，浅色和暗色其实对应两种完全不同的阅读环境。因此主题切换不只是美化设置，更是降低阅读疲劳的一部分。

## 评论区如何做到可选

评论能力通过 Waline 接入，但有两个前提：

1. 环境变量里配置了服务端地址
2. 当前文章 frontmatter 没有把 `comments` 关掉

也就是说，评论区既是站点级开关，也是文章级开关。

```bash
PUBLIC_WALINE_SERVER_URL="https://your-waline-server.vercel.app"
```

## 交互增强应该遵守的边界

我更偏向下面这组原则：

- [x] 能提升阅读或导航效率
- [x] 失败时不影响基础内容可读
- [x] 不要求用户理解额外概念
- [x] 不因为“有动画”就牺牲信息层级

如果一个功能必须靠明显提示才能让用户知道它存在，那它很可能已经太重了。

## 这篇文章在 demo 里的位置

前几篇文章更多在证明内容结构、MDX 和双语路由；而这一篇存在的理由，是把“主题增强体验”这一层单独拎出来演示清楚。

你可以把它看成对下面这些功能的集中验收：

- 视图过渡
- 预取
- 代码复制
- 阅读进度
- 回到顶部
- 主题切换
- Waline 评论接入

这些都不应该成为站点主角，但它们组合起来，能让一个长期写作主题真正顺手。
